# 高级前端面试题（含 React、AI、情景题）

## 使用说明

这份题库适用于高级前端、前端负责人、偏平台化/工程化/AI 应用方向岗位。
每道题都包含：

- 问题
- 标准答案
- 可追问点

建议面试时不要只看候选人是否“答对”，更要看：

- 是否能给出清晰的取舍
- 是否能结合真实项目场景
- 是否能识别边界条件和风险
- 是否具备工程落地能力，而不是只会背概念

## 一、架构与工程能力

### 1. 你如何判断一个前端项目是否已经“复杂到需要分层架构”？

**标准答案：**

通常看四个信号：

1. 业务模块之间开始相互耦合，改一个模块容易影响另一个模块。
2. 页面中同时承担视图渲染、状态管理、接口编排、权限判断、埋点等多种职责。
3. 多人并行开发时频繁出现冲突，或者相同逻辑在多个页面重复实现。
4. 新需求接入成本明显升高，开发者需要先理解很多历史代码才敢动。

这时应做分层，比如：

- `view`：只负责展示和交互
- `hooks/service`：封装业务逻辑和状态编排
- `api`：统一接口层
- `model/store`：共享状态
- `utils`：纯工具函数

核心目的不是“看起来高级”，而是降低变更成本和耦合度。

**可追问点：**

- 什么时候不适合过度分层
- 如何避免“分层过细导致跳转成本高”

### 2. 情景题：一个老项目首页首屏渲染要 7 秒，业务又不允许大改，你会怎么拆解优化？

**标准答案：**

先量化，再分治，不能上来就猜。

第一步：定位瓶颈。

- 用 Performance、Lighthouse、Web Vitals 看是网络慢、JS 执行慢、渲染慢还是接口慢。
- 区分 FCP、LCP、TTI、CLS 哪个指标最差。

第二步：按收益优先级优化。

- 如果是包太大：做路由级/组件级拆包，延迟加载低优先级模块。
- 如果是首屏接口慢：合并不必要请求、并发请求、做骨架屏、缓存静态数据。
- 如果是主线程阻塞：减少初始化逻辑，推迟埋点、编辑器、图表、复杂计算。
- 如果是渲染慢：减少不必要的重渲染，拆大组件，虚拟列表。
- 如果是资源慢：CDN、压缩、图片格式优化、preload 关键资源。

第三步：验证收益。

- 每次只做一类改动，确保优化可归因。
- 用真实用户监控而不是只看本地开发机。

在“不允许大改”的前提下，最现实的方案通常是：首屏拆包 + 接口并发 + 延迟初始化 + 缓存。

**可追问点：**

- 哪些优化对首屏有帮助，哪些只是“自我感动”
- 如何避免为了 Lighthouse 分数破坏真实体验

### 3. 你如何设计一个前端公共组件库，避免后期难维护？

**标准答案：**

组件库难维护通常不是因为组件少，而是因为边界不清晰。

设计原则：

- 组件只解决共性问题，不直接耦合具体业务字段。
- API 要稳定、可预测，避免过多布尔参数。
- 样式覆盖机制要提前设计好，比如 token、className、style、slots。
- 文档、示例、边界说明必须跟上。
- 对外暴露能力时，优先暴露组合能力而不是硬编码行为。

还要控制扩展方式：

- 通用组件进入组件库
- 半通用组件放业务域
- 一次性组件不要进组件库

否则组件库会变成“公共垃圾场”。

**可追问点：**

- 如何处理 breaking change
- 如何做版本管理和灰度发布

### 4. 情景题：你接手一个 React 项目，代码里到处是 `useEffect + setState + 请求`，状态混乱，你怎么收敛？

**标准答案：**

不能一次性重写，要分层治理。

我会先把问题分成三类：

- 数据获取逻辑混乱
- 派生状态过多
- Effect 承担了不该承担的逻辑

治理策略：

1. 把“请求数据”与“视图状态”拆开。
2. 能由 props 或 store 推导出的值，不再额外 `setState`。
3. 把副作用按职责拆分，不要一个 `useEffect` 做五件事。
4. 把重复请求逻辑收敛到自定义 hooks 或数据层。
5. 对复杂页面引入明确状态机或 reducer，避免散落的布尔值。

如果项目允许，我会逐步引入更稳定的数据管理方案，比如 React Query / SWR / Zustand / Redux Toolkit，而不是继续堆 `useEffect`。

**可追问点：**

- 哪类 `useEffect` 应该删除
- 如何判断某个 state 是冗余的

## 二、React 核心与进阶

### 5. React 中什么情况下不应该使用 `useMemo` 和 `useCallback`？

**标准答案：**

很多团队把它们当性能银弹，这是错的。

不该用的场景：

- 计算本身很轻，缓存收益低于维护成本。
- 组件本来就不频繁渲染。
- 只是为了“消除 ESLint 警告”而包一层。
- 依赖项复杂，反而更容易引入闭包问题和错误缓存。

应该先问两个问题：

1. 这个值/函数是否真的导致了昂贵计算或子组件无意义重渲染？
2. 缓存成本是否低于重复计算成本？

如果回答不明确，先别加。

**可追问点：**

- React Compiler 出现后如何看待这两个 Hook
- `memo`、`useMemo`、`useCallback` 的关系

---

### 6. 情景题：一个表单页面输入很卡，候选人说“加 useMemo 就好了”，你怎么看？

**标准答案：**

这是典型的拍脑袋优化。

输入卡顿的可能原因很多：

- 输入时触发了整个页面重渲染
- 某些昂贵组件跟着一起更新
- 大量同步校验逻辑阻塞主线程
- 富文本、图表、表格等重型组件没有隔离
- 状态设计不合理，局部输入导致全局状态更新

正确做法是：

1. 用 React DevTools Profiler 看是谁在重渲染。
2. 找到高频更新路径。
3. 做状态下沉或局部化。
4. 重型组件延迟更新，必要时用 `useDeferredValue` 或 `startTransition`。
5. 校验逻辑防抖或异步化。

`useMemo` 只有在“明确识别出某个昂贵计算”后才有意义。

**可追问点：**

- `useDeferredValue` 的适用场景
- 如何处理表单和预览联动卡顿

### 7. React 中如何理解“状态最小化”？

**标准答案：**

状态最小化的核心是：只保存不能被推导出来的数据。

比如：

- `fullName` 可以由 `firstName + lastName` 推导，不应单独存 state
- `filteredList` 可以由 `list + keyword` 推导，不应长期存 state
- `isEmpty` 可以由 `list.length === 0` 推导

这样做的好处：

- 避免多份状态不一致
- 降低更新链条复杂度
- 减少 bug 和 effect 数量

前端复杂页面里大量 bug，本质上都是“重复状态”造成的。

**可追问点：**

- 什么时候派生值也值得缓存
- 什么时候需要服务端状态和本地状态分离

---

### 8. 情景题：React 列表页有 2 万条数据，筛选和滚动都卡，你会怎么做？

**标准答案：**

这类问题通常要同时处理“渲染量”和“计算量”。

方案：

- 列表渲染层用虚拟列表，只渲染可视区。
- 筛选逻辑如果重，放到 Web Worker 或至少做防抖。
- 筛选输入和列表渲染分离，避免每次键入都同步阻塞。
- 单行组件做稳定 key，避免不必要卸载重建。
- 如果有展开、选中、编辑状态，要注意状态结构不能导致整表重渲染。

如果数据确实很大，还要反问业务：为什么前端要一次持有 2 万条？很多时候应该分页、游标加载或服务端筛选。

**可追问点：**

- 虚拟列表的坑
- 可变高度列表怎么处理

---

### 9. `useEffect`、`useLayoutEffect`、事件回调的职责边界是什么？

**标准答案：**

- 事件回调：处理用户动作后的同步业务逻辑
- `useEffect`：处理不阻塞浏览器绘制的副作用，比如请求、订阅、日志
- `useLayoutEffect`：处理需要在浏览器绘制前读取/修正布局的逻辑

原则上：

- 能放事件里就别塞进 effect
- 能放 `useEffect` 就别用 `useLayoutEffect`
- 不要把“状态同步另一个状态”滥用成 effect

`useLayoutEffect` 用多了会阻塞渲染，影响性能。

**可追问点：**

- 为什么很多 effect 应该被删除
- SSR 下 `useLayoutEffect` 的问题

## 三、情景题：线上稳定性与故障处理

### 10. 情景题：一个版本上线后，白屏率突然升高，你会如何排查？

**标准答案：**

先止血，再定位，再复盘。

止血：

- 如果有灰度或回滚能力，先降低影响面。
- 检查是否是静态资源 404、CDN 缓存错乱、接口鉴权失败导致。

定位：

- 看前端监控：错误栈、浏览器分布、页面分布、版本号分布。
- 看是否集中在某个 chunk 加载失败、某个浏览器特性不兼容、某个接口返回格式变化。
- 对比本次发布改动是否影响路由、初始化、权限、动态 import。

复盘：

- 补错误边界
- 补关键链路监控
- 补发布前 smoke test
- 对资源加载失败设计兜底重试

白屏处理能力是高级前端和普通前端的明显分水岭。

**可追问点：**

- chunk load error 怎么兜底
- 如何区分前端 bug 和后端变更导致

---

### 11. 情景题：接口偶发超时，用户频繁重复点击提交，结果出现重复下单，你怎么处理？

**标准答案：**

这是前后端协同问题，前端不能只靠“按钮置灰”。

前端侧：

- 提交中禁用按钮
- 防重复提交
- 请求超时给出明确反馈
- 对同一业务操作生成 request id

后端侧：

- 幂等设计
- 基于业务主键或 token 去重

高级答案不应该只说“按钮 loading 一下”，那只是 UX 层面的缓解，不是数据一致性的根本解法。

**可追问点：**

- 幂等 token 怎么设计
- 重试和重复提交的边界

---

### 12. 情景题：埋点 SDK 导致页面变卡，但业务又要求不能去掉埋点，你怎么办？

**标准答案：**

要做的是“降低埋点对主链路的干扰”，不是二选一。

做法：

- 延迟初始化非关键埋点
- 批量上报，减少频繁网络请求
- 对高频事件做节流/采样
- 复杂计算放 Worker 或空闲时执行
- 把与首屏无关的埋点下沉到可交互之后

同时要和业务统一口径：

- 哪些事件必须 100% 上报
- 哪些可以采样
- 哪些可以异步补偿

**可追问点：**

- 埋点 SDK 设计原则
- `requestIdleCallback` 的使用边界

## 四、AI 使用与 AI 前端应用

### 13. 你如何看待“AI 会不会替代前端”？

**标准答案：**

AI 会替代低价值、重复型、模板化的前端工作，但不会替代高质量工程判断。

更准确地说，AI 会重塑前端的工作结构：

- 模板代码、样板代码生成会越来越便宜
- 组件拼装和简单页面生成会被弱化
- 工程治理、性能优化、架构设计、复杂交互、业务抽象会更重要

未来高级前端的竞争力，不是“会不会写按钮”，而是：

- 能否把复杂问题结构化
- 能否定义边界和约束
- 能否把 AI 生成的结果变成可维护的生产代码

**可追问点：**

- 初级前端的成长路径会怎么变
- 团队招聘标准会如何变化

---

### 14. 情景题：团队开始大规模使用 AI 写代码，作为高级前端你最担心什么？

**标准答案：**

我最担心的不是“写错一行代码”，而是错误被规模化复制。

主要风险：

- 代码风格表面统一，但架构逐渐失控
- 团队成员不理解生成代码，后续维护困难
- 安全、性能、异常处理、边界条件缺失
- 旧模式被 AI 放大，技术债积累更快

应对策略：

- 明确 AI 可生成什么、不可生成什么
- 建立代码审查基线：状态管理、异常处理、测试、埋点、安全
- 关键模块必须人工设计骨架，AI 只能填充局部实现
- 对 AI 输出建立模板化 review checklist

**可追问点：**

- 哪些模块最不适合直接让 AI 生成
- 如何衡量 AI 对团队效率的真实提升

---

### 15. 情景题：让你设计一个“AI 助手嵌入业务后台”的前端方案，你会考虑哪些点？

**标准答案：**

不能只做一个聊天框，要按产品能力设计。

前端需要考虑：

- 会话管理：新建会话、历史会话、上下文隔离
- 流式输出：打字机效果、断流重试、取消生成
- 指令编排：普通问答、工具调用、表单填充、批量操作
- 权限控制：谁能问、谁能执行、谁能看到结果
- 结果可信度：引用来源、结构化输出、风险提示
- 可操作闭环：回答后能不能直接触发业务动作

如果是企业后台，AI 不应该只是“会说话”，而应该能嵌入现有工作流。

**可追问点：**

- SSE、WebSocket、长轮询怎么选
- 如何处理多轮上下文和成本控制

---

### 16. 情景题：一个 AI 对话页面经常出现“上一轮回答把下一轮顶掉”、“消息乱序”、“重复渲染”问题，你怎么设计状态？

**标准答案：**

这本质上是异步流式状态管理问题。

设计要点：

- 每条消息有稳定 id
- 每次生成任务有 request id / run id
- “用户消息”、“占位消息”、“流式增量”、“最终完成态”要区分
- 增量内容基于消息 id 合并，而不是简单 append
- 当前进行中的请求必须可取消，新的请求要么串行，要么显式并发隔离

不要只用一个 `messages` 数组粗暴覆盖，否则流式场景一定乱。

**可追问点：**

- 如何处理断流恢复
- 如何处理多 tab 会话一致性

---

### 17. 情景题：AI 返回的内容需要渲染 Markdown、代码块、表格、甚至可执行操作，你如何保证安全？

**标准答案：**

默认不可信。

必须考虑：

- Markdown 渲染要做 XSS 过滤
- HTML 默认禁用或严格白名单
- 链接要加安全属性
- 不允许模型直接返回可执行脚本
- 所有“执行类操作”必须走显式确认和权限校验

如果有“AI 生成 SQL / 命令 / 配置”能力，前端展示时也只能作为建议，真正执行必须经过服务端校验和权限控制。

**可追问点：**

- Markdown 渲染常见攻击点
- 为什么“只前端过滤”不够

---

### 18. 你在实际开发里如何正确使用 AI，而不是变成依赖 AI？

**标准答案：**

我的原则是“AI 提速，人类定边界”。

我会让 AI 做：

- 样板代码生成
- 多方案草稿
- 测试用例补全
- 文档初稿
- 重复性重构

我不会直接把 AI 当最终答案的场景：

- 架构边界定义
- 核心状态流设计
- 安全策略
- 性能瓶颈判断
- 线上故障分析

真正高效的方式不是“全交给 AI”，而是让 AI 处理低杠杆工作，人负责高杠杆判断。

**可追问点：**

- 如何给 AI 提高质量 prompt
- 如何审查 AI 生成代码

---

## 五、React + AI 复合情景题

### 19. 情景题：你要做一个“AI 驱动的表单填写助手”，用户一句话输入后，页面自动帮他填多个字段。你怎么设计？

**标准答案：**

关键点不是“填上去”，而是“可控、可审计、可纠错”。

设计上我会这样拆：

- 用户自然语言输入
- AI 返回结构化字段建议，而不是直接改 DOM
- 前端把建议映射到表单状态
- 对有风险的字段做高亮和人工确认
- 保留“接受全部 / 拒绝全部 / 逐项采纳”机制

原因是：AI 的输出不是事实，只是建议。直接覆盖用户输入会破坏信任。

**可追问点：**

- 怎么定义字段置信度
- 用户修改后的反馈是否回流模型

---

### 20. 情景题：一个 React 后台准备接入 AI 搜索，但现有页面有复杂筛选、分页、权限和缓存，你怎么融合？

**标准答案：**

AI 搜索不能绕过原有业务模型。

我会把它设计成“双通道检索”：

- 结构化筛选依然走原有查询链路
- AI 搜索负责自然语言解析、召回建议、补充解释

前端侧要做：

- AI 查询条件结构化展示
- 允许用户回退成普通筛选条件
- 搜索结果显示来源和命中依据
- 明确区分“模型理解”与“系统真实数据”

否则 AI 搜索很容易变成一个不可信黑盒。

**可追问点：**

- 如何做搜索解释性
- AI 理解错了如何回退

---

### 21. 情景题：在 React 项目里做 AI 流式回复时，页面切换后请求还在继续，导致内存泄漏和状态报警，你怎么处理？

**标准答案：**

要做请求生命周期管理。

方案：

- 为每次流式请求建立 `AbortController`
- 页面卸载或会话切换时主动 abort
- store 中标记当前激活会话，返回数据时校验归属
- 不在卸载组件上继续 setState

如果是全局会话系统，就不要把关键流状态只放在页面组件本地 state 中，而应该放在更稳定的数据层。

**可追问点：**

- SSE 如何取消
- WebSocket 如何做消息归属隔离

## 六、团队协作与技术判断

### 22. 你如何判断一个候选人是真的“高级前端”，还是只是年限长？

**标准答案：**

我主要看四点：

- 是否能把问题结构化，而不是只会罗列名词
- 是否能谈清楚真实项目里的取舍
- 是否能识别风险、边界和失败场景
- 是否能把方案落到工程层面

高级前端不是“知道很多 API”，而是能在复杂约束下给出靠谱方案。

**可追问点：**

- 你最看重哪类项目经历
- 如何识别“包装型候选人”

---

### 23. 情景题：产品要你三天内上一个复杂需求，你觉得技术方案风险很高，怎么沟通？

**标准答案：**

不能只说“做不了”，而要给选择题。

沟通方式：

- 先讲目标和约束
- 再讲风险点：时间、质量、兼容性、可维护性
- 然后给分级方案：快速版、稳妥版、完整版
- 明确每个方案的收益和代价

高级工程师的价值之一，就是把技术问题翻译成业务可以决策的信息。

**可追问点：**

- 如果产品坚持最快方案怎么办
- 如何记录技术债并推动偿还

### 24. 你会如何构建高级前端团队在 AI 时代的能力模型？

**标准答案：**

我会把能力分成四层：

1. 基础编码能力：React、TypeScript、工程化、性能、调试。
2. 复杂系统能力：状态建模、架构分层、稳定性、可观测性。
3. 业务抽象能力：把复杂流程沉淀成可复用能力。
4. AI 协同能力：高质量提问、结果审查、工具链接入、工作流重构。

未来优秀前端团队不是“大家都会用 AI”，而是知道：

- 什么时候该让 AI 做
- 什么时候必须人工主导
- 如何把 AI 纳入工程体系而不是游离在体系外

**可追问点：**

- 如何培训团队
- 如何定义 AI 使用规范

## 七、面试官建议

### 1. 不要只问概念题

高级前端最应该多问情景题、故障题、取舍题。

### 2. 追问“为什么”

候选人说用了某个方案，不够。
要追问：

- 为什么这么选
- 代价是什么
- 有什么替代方案
- 为什么没选别的

### 3. 关注“真实复杂度”

如果候选人的答案永远没有监控、异常、边界、协作、成本，那大概率只是会写功能，不是真正的高级工程师。

## 八、快速评分标准

### A 档

- 能讲清楚架构、性能、稳定性、工程治理
- 情景题有实战感
- AI 相关回答不空泛，能落到产品和工程

### B 档

- React 和工程化基础不错
- 能说出一些方案，但取舍不够深入
- AI 认知停留在工具使用层

### C 档

- 只会背概念
- 情景题回答泛泛
- 说不出真实项目中的问题和判断依据
